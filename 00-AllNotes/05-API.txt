1. INTRODUCTION TO APIs
================================================================================

What is an API?
---------------
API stands for Application Programming Interface. It's a contract between different software components that defines how they communicate with each other.

Think of it as a restaurant menu:
- **Menu** = API (what you can request)
- **Dish name** = Endpoint (what you want)
- **Your order** = Request (how you ask for it)
- **Food served** = Response (what you get back)

Key Characteristics:
- **Interface**: Set of rules and protocols
- **Abstraction**: Hides implementation details
- **Access Control**: Determines who can access what
- **Communication Format**: Structured data exchange

Why Do We Need APIs?
--------------------
1. **Decoupling**: Frontend and backend can evolve independently
2. **Code Reusability**: One API, multiple clients (web, mobile, desktop)
3. **Security**: Controlled, authenticated access to resources
4. **Scalability**: Can implement caching, load balancing, rate limiting
5. **Integration**: Connect different systems and services
6. **Flexibility**: Different tech stacks can work together
7. **Standardization**: Common interface for consistent data access

Real-World Examples:
- Twitter API: Access tweets, users, trends programmatically
- Google Maps API: Embed maps in your application
- Payment APIs: Stripe, PayPal for processing payments
- Weather APIs: Get weather data for your location
- Email APIs: SendGrid for sending emails

2. BEFORE MODERN APIs - THE EVOLUTION
================================================================================

How Did Applications Communicate Before?

1. **Direct Database Access (1980s-1990s)**
   - Frontend connected directly to database
   - Pros: Simple, fast
   - Cons: Security nightmare, tight coupling, no abstraction
   - Example: VB application directly querying SQL Server

2. **Screen Scraping (1990s)**
   - Parse HTML from websites to extract data
   - Pros: Works with any website
   - Cons: Brittle, slow, unethical
   - Example: Scraping HTML with regex

3. **CORBA/DCOM (1990s)**
   - Common Object Request Broker Architecture
   - Binary protocol for distributed objects
   - Pros: Language-independent, powerful
   - Cons: Complex, difficult to implement, poor firewall support

4. **SOAP (1998+)**
   - Simple Object Access Protocol (ironically not simple)
   - XML-based messaging
   - Heavy, rigid, verbose
   - Still used in enterprise/banking

5. **FTP Batch Transfers (1980s-2000s)**
   - Manual file uploads/downloads
   - Scheduled batch jobs
   - Pros: Works for bulk data
   - Cons: Not real-time, manual process

6. **Proprietary Solutions (2000s)**
   - Companies built custom protocols
   - No standardization
   - Hard to integrate

Modern APIs emerged to solve these problems with standardization and simplicity.

3. REST API - THE MOST COMMON API STYLE
================================================================================

What is REST?
-------------
REST = Representational State Transfer

A set of principles for designing networked applications based on HTTP.

REST Principles (Constraints):

1. **Client-Server Architecture**
   - Clear separation between client and server
   - Can evolve independently

2. **Statelessness**
   - Each request contains all information needed
   - Server doesn't store client session state
   - Better scalability

3. **Uniform Interface**
   - Consistent way of accessing resources
   - Standard HTTP methods
   - Resources identified by URLs

4. **Resource-Oriented**
   - Everything is a resource (users, posts, comments)
   - Resources have unique URIs
   - Resources have representations (JSON, XML)

5. **Cacheable**
   - Responses can be cached
   - Improves performance

6. **Layered System**
   - Communication goes through multiple layers
   - Each layer doesn't know about others

REST API Design:

**Resources and Endpoints:**
Resources are nouns, not verbs:
```
/api/users              # Collection of users
/api/users/123          # Specific user
/api/users/123/posts    # User's posts
/api/users/123/posts/5  # Specific post by user
```

**HTTP Methods (Verbs):**
```
GET    /api/users          # Get all users
GET    /api/users/123      # Get user 123
POST   /api/users          # Create new user
PUT    /api/users/123      # Replace user 123
PATCH  /api/users/123      # Partial update user 123
DELETE /api/users/123      # Delete user 123
```

Detailed Method Explanations:

**GET** (Retrieve):
- Safe: Doesn't change server state
- Idempotent: Multiple calls return same result
- Cacheable: Results can be cached
- Example: `GET /api/users/123` returns user data

**POST** (Create):
- Not safe: Changes server state
- Not idempotent: Multiple calls create multiple resources
- Not cacheable
- Example: `POST /api/users` with body `{name:'John', email:'john@example.com'}`

**PUT** (Replace):
- Not safe: Changes server state
- Idempotent: Multiple calls with same data = same result
- Full replacement of resource
- Example: Replace entire user 123 with new data

**PATCH** (Partial Update):
- Not safe: Changes server state
- Not always idempotent (depends on operation)
- Partial modification
- Example: UpdateUser 123's email only

**DELETE** (Remove):
- Not safe: Changes server state
- Idempotent: Multiple calls + same result
- Removes resource
- Example: `DELETE /api/users/123`

**HTTP Status Codes:**

Success/Information:
- **200 OK**: Request succeeded
- **201 Created**: Resource created successfully
- **204 No Content**: Successful, no response body
- **206 Partial Content**: Partial response (pagination)

Redirection:
- **301 Moved Permanently**: Resource moved permanently
- **302 Found**: Temporary redirect
- **304 Not Modified**: Use cached version

Client Errors:
- **400 Bad Request**: Invalid request format
- **401 Unauthorized**: Authentication required
- **403 Forbidden**: Authenticated but not allowed
- **404 Not Found**: Resource doesn't exist
- **409 Conflict**: Request conflicts with state
- **422 Unprocessable Entity**: Validation failed

Server Errors:
- **500 Internal Server Error**: Server error
- **503 Service Unavailable**: Server temporarily down

REST API Example (Node.js/Express):
```javascript
const express = require('express');
const app = express();
app.use(express.json());

let users = [
  { id: 1, name: 'Alice', email: 'alice@example.com' },
  { id: 2, name: 'Bob', email: 'bob@example.com' }
];

// GET all users
app.get('/api/users', (req, res) => {
  res.json(users);
});

// GET specific user
app.get('/api/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).json({ error: 'User not found' });
  res.json(user);
});

// POST create user
app.post('/api/users', (req, res) => {
  const newUser = {
    id: users.length + 1,
    name: req.body.name,
    email: req.body.email
  };
  users.push(newUser);
  res.status(201).json(newUser);
});

// PUT replace user
app.put('/api/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).json({ error: 'User not found' });
  
  user.name = req.body.name;
  user.email = req.body.email;
  res.json(user);
});

// PATCH partial update
app.patch('/api/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).json({ error: 'User not found' });
  
  if (req.body.name) user.name = req.body.name;
  if (req.body.email) user.email = req.body.email;
  res.json(user);
});

// DELETE user
app.delete('/api/users/:id', (req, res) => {
  const userIndex = users.findIndex(u => u.id === parseInt(req.params.id));
  if (userIndex === -1) return res.status(404).json({ error: 'User not found' });
  
  const deletedUser = users.splice(userIndex, 1);
  res.json({ message: 'User deleted', user: deletedUser[0] });
});

app.listen(3000);
```

REST Query Parameters:

```javascript
// GET /api/users?page=1&limit=10&sort=name
app.get('/api/users', (req, res) => {
  const page = req.query.page || 1;
  const limit = req.query.limit || 10;
  const sort = req.query.sort || 'id';
  
  // Implement pagination and sorting logic
  res.json(users);
});
```

REST Response Format Best Practices:
```json
{
  "success": true,
  "status": 200,
  "message": "User retrieved successfully",
  "data": {
    "id": 1,
    "name": "Alice",
    "email": "alice@example.com"
  }
}
```

Error Response Format:
```json
{
  "success": false,
  "status": 404,
  "message": "User not found",
  "error": {
    "code": "USER_NOT_FOUND",
    "details": "No user with ID 999 exists"
  }
}
```

4. GRAPHQL - QUERY YOUR EXACT DATA NEEDS
================================================================================

What is GraphQL?
----------------
A query language and runtime for APIs developed by Facebook.

Key Idea:
"Ask for exactly what you need, get exactly that, nothing more, nothing less."

GraphQL vs REST Comparison:

REST Problems GraphQL Solves:
1. **Over-fetching**: REST returns entire resource, you may need only few fields
2. **Under-fetching**: REST may require multiple requests to get related data
3. **Versioning Issues**: REST requires API versioning (/v1/, /v2/)
4. **Strict Response Format**: Can't customize response structure

Single Endpoint:
- REST: Multiple endpoints (/users, /posts, /comments)
- GraphQL: Single endpoint (/graphql)

GraphQL Query Example:
```graphql
# Request only name and email, not all user fields
query {
  user(id: 1) {
    name
    email
  }
}
```

REST equivalent would need multiple calls or return extra fields.

GraphQL Mutation Example (Like POST/PUT/DELETE):
```graphql
mutation {
  createUser(name: "John", email: "john@example.com") {
    id
    name
    email
  }
}
```

GraphQL Schema:
```graphql
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  content: String!
  author: User!
}

type Query {
  user(id: ID!): User
  allUsers: [User!]!
  post(id: ID!): Post
}

type Mutation {
  createUser(name: String!, email: String!): User!
  updateUser(id: ID!, name: String, email: String): User!
  deleteUser(id: ID!): Boolean!
}
```

GraphQL Advantages:
- Strongly typed schema
- Automatic documentation
- Tool support and IDE integration
- Resolves N+1 query problem with proper implementation
- No versioning needed

GraphQL Disadvantages:
- Learning curve (schema definition)
- More complex server implementation
- File upload handling is awkward
- Caching is harder (everything is POST)
- Query complexity can lead to performance issues

Node.js GraphQL Example:
```javascript
const express = require('express');
const { graphqlHTTP } = require('express-graphql');
const { buildSchema } = require('graphql');

const schema = buildSchema(`
  type User {
    id: ID!
    name: String!
    email: String!
  }

  type Query {
    user(id: ID!): User
    allUsers: [User!]!
  }

  type Mutation {
    createUser(name: String!, email: String!): User!
  }
`);

let users = [
  { id: '1', name: 'Alice', email: 'alice@example.com' },
  { id: '2', name: 'Bob', email: 'bob@example.com' }
];

const root = {
  user: ({ id }) => users.find(u => u.id === id),
  allUsers: () => users,
  createUser: ({ name, email }) => {
    const newUser = { id: String(users.length + 1), name, email };
    users.push(newUser);
    return newUser;
  }
};

const app = express();
app.use('/graphql', graphqlHTTP({
  schema,
  rootValue: root,
  graphiql: true // Interactive GraphQL explorer
}));

app.listen(3000);
```

5. gRPC - HIGH-PERFORMANCE MICROSERVICES
================================================================================

What is gRPC?
-------------
gRPC = Google Remote Procedure Call

A high-performance, open-source RPC framework developed by Google.

Key Features:
- Uses HTTP/2 for multiplexing
- Protocol Buffers for serialization (binary, compact)
- Strong typing
- Built-in streaming support

gRPC vs REST:

| Feature          | gRPC           | REST          |
|------------------|----------------|---------------|
| Protocol         | HTTP/2         | HTTP/1.1      |
| Serialization    | Protobuf       | JSON/XML      |
| Message Size     | Smaller        | Larger        |
| Speed            | Faster         | Slower        |
| Browser Support  | Limited        | Full          |
| Human Readable   | No (binary)    | Yes (JSON)    |
| Use Case         | Microservices  | Public APIs   |

When to Use gRPC:
- Internal microservices communication
- High-performance requirements
- Need for streaming
- Real-time applications

When NOT to Use gRPC:
- Public APIs (browser clients)
- Simple CRUD operations
- When JSON/REST perfectly fits

gRPC Protocol Buffer Definition:
```protobuf
syntax = "proto3";

message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

service UserService {
  rpc GetUser(GetUserRequest) returns (User) {}
  rpc CreateUser(CreateUserRequest) returns (User) {}
  rpc ListUsers(Empty) returns (stream User) {}
}
```

6. SOAP - ENTERPRISE MESSAGING
================================================================================

What is SOAP?
-------------
SOAP = Simple Object Access Protocol (ironic - it's quite complex)

XML-based protocol for web services communication.

SOAP Characteristics:
- XML-based (verbose)
- Stateful (unlike REST)
- Built-in error handling (Faults)
- WS-Security for encryption
- WSDL (Web Services Description Language) for interface

When Still Used:
- Banking and financial systems
- Enterprise applications
- Systems requiring strict standards
- Legacy systems

SOAP Request Example:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap-envelope/">
  <soap:Body>
    <GetUser xmlns="http://example.com/service">
      <userId>123</userId>
    </GetUser>
  </soap:Body>
</soap:Envelope>
```

Why REST Replaced SOAP:
- Simpler to understand and implement
- Built on HTTP standards
- JSON is lighter than XML
- Easier for web development
- Better browser support

7. WEBSOCKETS - REAL-TIME COMMUNICATION
================================================================================

What are WebSockets?
--------------------
Protocol for persistent, bidirectional communication over a single TCP connection.

HTTP vs WebSocket:
- HTTP: Request-response, stateless
- WebSocket: Persistent connection, real-time bidirectional

When to Use WebSockets:
- Chat applications
- Live notifications
- Real-time multiplayer games
- Live dashboards
- Collaborative editing

WebSocket Lifecycle:
1. HTTP upgrade request
2. Handshake successful
3. Persistent connection established
4. Bidirectional messaging
5. Connection close

Socket.IO Example (Popular WebSocket library):
```javascript
const express = require('express');
const http = require('http');
const socketIo = require('socket.io');

const app = express();
const server = http.createServer(app);
const io = socketIo(server);

io.on('connection', (socket) => {
  console.log('User connected:', socket.id);

  // Listen for chat messages
  socket.on('chat message', (msg) => {
    io.emit('chat message', {
      userId: socket.id,
      message: msg
    });
  });

  // Handle disconnect
  socket.on('disconnect', () => {
    console.log('User disconnected:', socket.id);
  });
});

server.listen(3000);
```

8. API GATEWAY PATTERN
================================================================================

What is an API Gateway?
-----------------------
A server that sits between clients and multiple backend services.

Responsibilities:
- **Routing**: Direct requests to appropriate services
- **Authentication**: Verify user identity
- **Authorization**: Check permissions
- **Rate Limiting**: Prevent abuse
- **Logging & Monitoring**: Track API usage
- **Request/Response Transformation**: Modify messages
- **Load Balancing**: Distribute across services
- **Caching**: Store responses

API Gateway Architecture:

```
Client → API Gateway → Service 1
                     → Service 2
                     → Service 3
```

Benefits:
- Single entry point for clients
- Centralized security
- Service decoupling
- Easy to add/modify services
- Simplified versioning

Popular API Gateways:
- Kong
- AWS API Gateway
- Azure API Management
- Zuul (Netflix)
- Traefik

Example (Kong Configuration):
```yaml
services:
  - name: user-service
    host: user-service
    port: 3000
    routes:
      - name: users
        paths:
          - /users

  - name: order-service
    host: order-service
    port: 4000
    routes:
      - name: orders
        paths:
          - /orders

plugins:
  - name: jwt
    config:
      key_claim_name: iss
```

9. API DESIGN BEST PRACTICES
================================================================================

Version Your APIs:
- Use URL versioning: `/api/v1/users`
- Or header versioning: `Accept: application/vnd.myapp.v1+json`
- Backward compatibility considerations

RESTful URL Patterns:
```
✓ GET    /api/v1/users
✓ GET    /api/v1/users/123
✗ GET    /api/getAllUsers          (verb in URL)
✗ GET    /api/v1/user              (singular for collection)
```

Pagination:
```
GET /api/users?page=1&limit=10
GET /api/users?offset=0&limit=10
GET /api/users?cursor=abc123&limit=10
```

Filtering, Sorting, Searching:
```
GET /api/users?status=active&role=admin
GET /api/users?sort=name&order=asc
GET /api/users?search=john
GET /api/users?search=john&filter[role]=admin&sort=-createdAt
```

Consistent Error Responses:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid request",
    "details": [
      { "field": "email", "message": "Invalid email format" }
    ]
  }
}
```

Documentation:
- Use OpenAPI/Swagger for documentation
- Include examples for each endpoint
- Explain error scenarios

Rate Limiting Headers:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1633024800
```

API Documentation Example (OpenAPI):
```yaml
openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
paths:
  /api/v1/users:
    get:
      summary: Get all users

        - name: page

          type: integer
          required: false

          description: List of users
          schema:
            type: array
            items:
              $ref: '#/components/schemas/User'
```

10. AUTHENTICATION & AUTHORIZATION IN APIs
================================================================================

Authentication (Proving Identity):
-----------------------------------

1. **API Key**
   - Simple but less secure
   - Send in header: `X-API-Key: your-api-key`
   ```javascript
   app.use((req, res, next) => {
     const apiKey = req.headers['x-api-key'];
     if (!isValidKey(apiKey)) return res.status(401).json({ error: 'Invalid API key' });
     next();
   });
   ```

2. **Basic Authentication**
   - Username:password in header
   - Must be HTTPS only
   - `Authorization: Basic base64(username:password)`

3. **Bearer Token (JWT)**
   - Most common modern approach
   - `Authorization: Bearer token`
   - Stateless, scalable
   ```javascript
   const jwt = require('jsonwebtoken');
   
   app.use((req, res, next) => {
     const token = req.headers.authorization?.split(' ')[1];
     if (!token) return res.status(401).json({ error: 'No token' });
     
     try {
       const decoded = jwt.verify(token, 'secret');
       req.user = decoded;
       next();
     } catch (err) {
       res.status(401).json({ error: 'Invalid token' });
     }
   });
   ```

4. **OAuth 2.0**
   - Delegated access
   - Third-party login (Google, GitHub)

Authorization (What You Can Access):
------------------------------------

```javascript
// Role-based access control
function authorize(roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

app.delete('/api/users/:id', authorize(['admin']), (req, res) => {
  // Only admins can delete
});
```

11. RATE LIMITING & THROTTLING
================================================================================

Why Rate Limit?
- Prevent abuse and DDoS attacks
- Ensure fair usage
- Protect resources

Strategies:

1. **Fixed Window**
   - Reset counter at fixed intervals
   - Simple but uneven distribution

2. **Sliding Window**
   - More sophisticated
   - Better distribution but complex

3. **Token Bucket**
   - Tokens for each request
   - Requests queue for tokens
   - Smooth rate limiting

4. **Leaky Bucket**
   - Smooths out bursts
   - Consistent output rate

Express Rate Limiting Example:
```javascript
const rateLimit = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // Limit each IP to 100 requests per windowMs
  message: 'Too many requests from this IP'
});

app.use(limiter);

// Stricter limit for login
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5 // Only 5 login attempts per 15 minutes
});

app.post('/api/login', loginLimiter, (req, res) => {
  // Login logic
});
```

12. API VERSIONING STRATEGIES
================================================================================

URL Path Versioning (Most Common):
```
/api/v1/users
/api/v2/users
```

Pros: Clear, easy to implement
Cons: Creates URL duplication

Header Versioning:
```
Accept: application/vnd.myapp.v2+json
```

Pros: Cleaner URLs
Cons: Less visible

Query Parameter Versioning:
```
/api/users?version=2
```

Pros: Simple
Cons: Non-standard

Versioning Strategy:
- New major version when breaking changes
- Deprecated old versions with notice
- Support multiple versions for transition period
- Provide migration guide

3. **Insomnia**
   - Similar to Postman

4. **REST Client** (VS Code Extension)
   - Test directly in editor

API Testing Best Practices:
- Test happy paths
- Test error scenarios (404, 500, etc.)
- Test validation errors
- Test authentication/authorization
- Test rate limiting
- Load testing

Unit Test Example:
```javascript
const request = require('supertest');
const app = require('./app');

describe('User API', () => {
  test('GET /api/users returns all users', async () => {
    const response = await request(app).get('/api/users');
    expect(response.status).toBe(200);
    expect(Array.isArray(response.body)).toBe(true);
  });

  test('POST /api/users creates user', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: 'John', email: 'john@example.com' });
    expect(response.status).toBe(201);
    expect(response.body.name).toBe('John');
  });
});
```

14. CHOOSING THE RIGHT API TYPE
================================================================================

Use REST When:
- Building public APIs
- Simple CRUD operations
- Browser clients needed
- Team familiar with REST

Use GraphQL When:
- Multiple clients with different data needs
- Complex, nested data structures
- Want to reduce over-fetching
- Can afford complexity

Use gRPC When:
- Internal microservices communication
- Performance is critical
- Need streaming
- All clients are programming languages

Use WebSockets When:
- Real-time bidirectional communication
- Real-time notifications
- Live collaboration features

Use SOAP When:
- Enterprise legacy systems
- Strict compliance requirements
- Need WS-Security
